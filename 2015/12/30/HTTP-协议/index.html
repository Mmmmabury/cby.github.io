HTTP 协议基础介绍

发表于 2015-12-30   |   分类于 Web协议   |  

The Hypertext Transfer Protocol (HTTP) is an application-level protocol for distributed, collaborative, hypermedia information systems.

HTTP 简述

什么是 HTTP

在信息时代，每天我们都能从各色网页中获取几乎无尽的信息，而这些网页其实可以看成是一个大的应用，称为万维网（World Wide Web，简称WWW）。对于很多人来说，因特网就是万维网，其实不是，万维网只是因特网的提供的一项服务。万维网的定义是：一个可通过互联网访问的由许多互相链接的超文本组成的系统。而互相链接的超文本就是用 HTTP 协议来传输的。

HTTP（Hypertext Transfer Protocol）是基于 TCP/IP 的应用层协议，用来分发和接收超媒体信息。那么，为什么 HTTP 的英文是超文本（hypertext）传输协议，而官方定义却是用超媒体（hypermedia）呢。这是一个历史问题，最早先是定义了超文本，后来更多种类的资源加入了，例如动画，视频。所以将超文本拓展，称为超媒体。通常我们可以将超文本等同于超媒体。

那什么是超媒体呢？超媒体（Hypermedia）是一种非线性的信息结构，也就是说它是网状的，信息之间是能进行交叉引用的，通常是用超链接进行互相引用。超媒体包括有图形，视频，音频，文本，超链接。常用的超媒体格式是 HTML（超文本标记语言）。

简而言之，因特网就像一个人类社会，万维网是就是基于人类社会的信息系统，由各色各样有用的资源（个人脑海中的信息）构成。在这个系统中，当你向我请求某一信息，例如“白云讲讲狼来了的故事”，这时我用约定俗成的语言——中文向你讲故事，你听懂了。这个约定俗成的语言便是 HTTP 协议。HTTP 协议中所有的标准都是为了让信息更可靠、高效、完整地传输。

HTTP 协议历史

HTTP 协议的历史最早能追溯到超文本的推出。在1945年，Vannervar Bush 出版《As We May Think》一书，书中假象了一种拓展存储器，称为 Memex。这是一种基于微缩胶卷的图书馆，可以根据交叉引用来播放图书和影片。受这个观点影响，Ted Nelson 在1965年提出了超文本的概念。随后 Tim Berners-Lee 和他的小伙伴们在欧洲核子能研究所（CERN）发明了 HTTP 用来传输超文本资源。并且在1989年提出了万维网（World Wide Web）。最早的 HTTP 协议只有一个方法 GET，它能向某一服务器请求页面，而从服务响应的通常是一个 HTML 页面。最早的拥有文档的 HTTP 协议版本是 HTTP/0.9(1991)，之后在1995年 Dave Raggett 建立了 HTTP 工作组，这个工作组的目的就是拓展 HTTP 协议的功能。下面列举了 HTTP 的不同版本：

  1. HTTP/0.9
    最早的原型，只支持 GET 方法，不支持 MIME 类型，各种 HTTP 首部等等。
  2. HTTP/1.0
    HTTP 添加很多方法，各种 HTTP 首部，以及对多媒体对象的处理。
  3. HTTP/1.0+
    1.0+是非官方的事实标准，主要因为当时 Web 服务飞快发展，各个服务器厂商都向 HTTP 中添加各种特性，其中有很多特性例如持久的长连接、虚拟主机的支持以及代理支持都被加入到 HTTP 中，这些非官方的 HTTP 拓展版本就被称为 HTTP/1.0+
  4. HTTP/1.1
    HTTP/1.1是当前主流的 HTTP 协议。完善了之前 HTTP 设计中的结构性缺陷，明确了语义，添加和删除了一些特性，支持更加复杂的的 Web 应用。HTTP/1.1由 RFC2616 定义。
  5. HTTP/2.0
    HTTP/2.0是最新的 HTTP 协议，已于2015年5月份正式发布， Chrome、 IE 以及 firefox 等宣布将支持或者已经支持 HTTP/2 协议。HTTP/2协议又花了性能并且对 HTTP/1.1兼容。

如何使用 HTTP

知道了 HTTP 是何物以及简单的历史，那么在现实中如何使用呢。想象下万维网是一个复杂的网状物，像下图：

它如此巨大，包含着海量的资源，为了从中找到一个资源，我们必须给每一个资源一个唯一的标识，所以我们有了统一资源标识符；除了得到资源，我们可能会需要删除资源，提交资源等等操作，所以我们有了 HTTP 方法；为了让交流的双方知道对方的状态，所有有了首部（head）和状态码。当然还有其他的规定。所有规定构成了 HTTP 的轮廓，并且使 HTTP 能够在现实中可靠顺利的使用。

资源的类型和位置

在万维网中拥有很多资源，所以需要为他们分配一个唯一的名称，用来标识该资源。一个资源需要标识它的类型和位置，通常我们使用 MIME 类型和 URI 来标识。

资源类型

万维网上有数千种不同的数据类型，HTTP 协议给每种数据都打上了名为 MIME type 的标签。最早电子邮件系统设计并使用了 MIME（Multipurpose Internet Mail Extension），用来解决不同邮件系统间报文兼容问题。后来被 HTTP 所采纳。MIME 类型是文本标记，可读性强，格式为 “父类型/子类型” ，例如：

text/html 文本父类型下的 html 类型
image/jpeg 照片类型下的 jpeg 类型

MIME type 描述了数据的格式，浏览器会由收到的 MIME type 标签来判断如何处理数据。更多MIME type 在这里。

资源的位置

URI（Uniform Resource Identifier）是统一资源标识符，它分为 URL（统一资源定位符） 和 URN（统一资源名）。URI 确定了资源的精确地址以及以何种方式去访问它

URL 一般分为三部分组成：

  • 方案（scheme）
    说明访问资源所使用的协议，在 HTTP 中通常就是（http://）

  • 主机
    网络中的主机地址

  • 路径
    指向了 Web 服务器中的某个资源

遵从 HTTP 协议的 URL 通常具有以下形式：
http:// hostname [:port] / path
例如：
http://www.baidu.com

而URN 现在还是在实验阶段，每一个 URN 都对应着唯一一个资源，不论资源在何处，都可以通过 URN 访问到。不过 URN 现在仍处于试验阶段，还未大范围使用。

HTTP 结构

如果说 HTTP 协议是因特网的信使，HTTP 报文就是包裹了。每一个在 HTTP 应用之间的数据块都是报文。
HTTP 报文可以分为两部分：请求报文和响应报文
报文是文本文件，而不是二进制代码，请求报文和响应报文的格式很相似，都分为三部分。

  1. 起始行
    起始行一般是报文的第一行。在请求报文中，说明要做什么；在响应报文中，说明出现了什么。一般包括 URI 、请求方法以及状态码

  2. 首部（head）字段
    首部字段是键值对来表示，两者之间用冒号来分隔，首部以一个空行结束

  3. 实体内容
    这里是 HTTP 传输的正文，HTTP 报文可以承载很多数字类型，例如图片、视频、HTML 文档等等

HTTP 请求报文

请求报文格式：

[方法] [资源位置] [HTTP版本]
[首部]
……

[主体]

  • 方法
    告诉服务要做什么，后面有详细介绍
  • 资源位置
    相对URL，告诉服务器资源位置
  • HTTP 版本
    告诉服务器 HTTP 版本，让服务器使用对应的版本逻辑，以防使用不兼容的方法
  • 首部
    见后面
  • 主体
    HTTP 传输的超媒体内容

HTTP 响应报文

响应报文格式：
[HTTP 版本] [状态码] [可读状态码也叫原因短语]
[首部]
……

[主体]

  • HTTP 版本
    告诉服务器 HTTP 版本，让服务器使用对应的版本逻辑，以防使用不兼容的方法
  • 状态码
    告诉客户端请求结果，见后
  • 原因短语
    状态码的文字描述
  • 首部
    见后
  • 主体
    内容

HTTP 方法

现在的 HTTP 方法有很多，但是在服务器中不需要全部实现，只要实现 GET 和 HEAD 方法就可以与 HTTP/1.1兼容了。

其中常用的有：GET，HEAD，POST，PUT

GET

GET 是最常见的方法，用于请求服务器发送某个资源。
GET 请求的报文例子：

服务器回复：

GET 请求是不带主体内容的，因为不需要资源来请求，而服务器的响应就带了内容。

HEAD

HEAD 方法也是请求服务器返回资源，只不过只要求返回 HEAD（首部）。HEAD 方法的作用有：

  1. 在不获取资源的情况下了解资源的情况
  2. 通过状态码看看某个对象是否存在
  3. 查看首部，测试资源是否被修改了（比如看 Content-length）

服务器遵循 HTTP/1.1规范就必须实现 HEAD 方法。

PUT

PUT 方法是向服务器提交文档。让服务用请求的主体部分来创建一个由所请求的 URL 命名的新文档，或者覆盖掉已有的文档。因为 PUT 会对服务器内容进行修改，所以一般会进行密码认证。

TRACE

TRACE 主要用于诊断。 一个客户端的请求需要穿过防火墙、代理、网关或者其他一些应用程序。每个中间节点都可能会修改原始的 HTTP 请求。而 TRACE 方法允许客户端查看 HTTP 请求最终变成了什么样子。TRACE 方法会在最后一程服务器发起一个”环回“诊断，服务器会返回一条 TRACE 响应，附带接收到的 HTTP 请求副本。也就是说 TRACE 请求中不能带有实体的主体部分，而 TRACE 响应的主体中带有服务器收到的请求的副本。

DELETE

DELETE 请服务器删除请求 URL 所指定的资源。但是服务器不一定会执行操作，并且 HTTP 规范中也允许服务器不通知客户端的情况撤回操作。

OPTIONS

OPTIONS 请求 Web 服务器，告知其支持的各种功能。

拓展方法

HTTP 有良好的拓展性，拓展方法是没有再规范中定义的，可以看成是开发者私有的方法。

HTTP 状态码

HTTP 状态码分成了五大类。

1XX信息性状态码(Informational)

1XX 信息性状态码,表示消息已经收到，请继续

2XX成功状态码(Successful)

2XX 成功状态码，这类状态码表示消息通常是成功的

状态码 消息 含义
200 OK 请求成功
201 Created 创建对象成功，比如说 POST 请求发送了一个资源，服务器成功创建
202 Accepted 请求被接受，例如发送 DELETE 请求，表示服务器已经接受到这个请求，但不一定已经执行
203 Non-Authoritative Information 说明实体首部包含的信息不是来自于源端服务器，而是来源于一个副本。
204 No Content 表示响应的报文中没有内容，只有首部和状态行。可以用于刷新页面的某一部分
205 Reset Content 主要用于浏览器的代码，负责告知浏览器清楚当前页面中的所有 HTML 表单元素
206 Partial Content 成功执行了 Range 请求。Range 请求是向服务器请求某个范围内的文档

3XX重定向状态码(Redirection)

重定向状态码，告知客户端使用替代位置来访问需要的资源。

状态码 消息 含义
300 Multiple Choices 客户端请求的 URL 指向多个资源
301 Moved Permanently 请到服务器返回的 URL 中请求资源
302 Found 与301类似，临时重定向
303 See Other 告知客户端应该使用另一个 URL 来获取资源，将非 GET 和 HEAD 转为 GET
304 Not Modified 带条件的重定向
305 Use Proxy 使用代理访问资源
307 Temporary Redirect 对于POST请求，表示请求还没有被处理，客户端应该向Location里的URI重新发起POST请求。

301、302的区别

301的含义是“永久重定向”，而302的含义是“临时重定向”
301 代表永久性转移是网页更改地址后对搜索引擎友好的最好方法。

302、303和307的区别

在 HTTP1.0 的时候，文档规定如果客户端发送非 GET 和 HEAD 方法到服务器，而从服务器接收到302时，需要向用户确定是否重新发送。但是很多浏览器却没有遵守文档，而是直接改为 GET 重新发送。
在 HTTP1.1 的时候，文档添加了 303，表示可以将非 GET 和 HEAD 方法直接转化为 GET 方法重定向访问，不需要询问用户。在这个时候，很多浏览器却没有改变，依然使用302，而且把它当做303使用，也就是说浏览器收到302，不询问用户直接就转为 GET 重定向访问了。后来引入了307（主要是对于 POST 方法定义的），表示当客户端的POST请求收到服务端307状态码响应时，需要跟用户询问是否应该在新URI上发起POST方法，也就是说，307是不会把 POST 转为 GET 的。
总的来说，303和307是302的细分，302是老的状态码，它只定义了重定向需要询问用户。303是定义了将 POST 直接转为 GET 再重定向访问，而307不会转为 GET，仍旧用 POST 访问，而且需要询问用户。

状态码 定义
302 对于所有方法，需要询问用户后再获取报文头部的 location 字段值，并重定向访问
303 对于非 GET 和 HEAD 方法，将其转化为 GET 方法，对 location 里的 URI 重新访问
307 对于 POST 方法，客户端向 location 中的 URI 重新发起 POST 请求，并且需要询问用户

4XX重定向状态码(Client Error)

4XX 客户端错误状态码
很多客户端错误码是直接由浏览器处理的，但是也有些是由服务器处理的，比如404

400 Bad request 错误的请求
401 Unauthorized 需要进行认证
403 Forbidden 服务器拒绝访问
404 Not Found 所请求的页面无法被访问
405 Method Not Allowed 请求的方法无法处理，或是服务器不支持改请求方法
406 Not Acceptable 表示客户端向服务器发送愿意接受的类型，若服务器没有该类型文件，返回406
407 Proxy Authentication Required 需要进行代理认证
408 Request Timeout 请求超时
409 Conflict 请求发生冲突
410 Gone 与404类似，只是服务器曾经拥有过此资源
411 Length Required 要求报文中包含 Content-Length 首部
412 Precondition Failed 客户端发起了条件请求，其中一个请求失败了
413 Request Entity Too Large 实体内容部分过大
414 Request URI Too Long URI 太长
415 Unsupported Media Type 服务器无法支持内容类型
416 Requested Range Not Satisfiable 请求范围无效
417 Expectation Failed 无法满足 Expect 期望

5XX服务器错误代码(Server Error)

5XX 服务器错误代码

500 Internal Server Error 服务器遇到错误
501 Not Implemented 客户端发送的请求超出了服务器能力范围
502 Bad Gateway 一般是代理服务器出问题
503 Service Unavailable 说明服务器现在无法提供服务，将来可以
504 Gateway Timeout 代理或者网关响应超时
505 HTTP Version Not Supported HTTP 版本不支持

  2. 响应首部。响应报文特有
  3. 实体首部。用于说明实体
  4. 拓展首部。非标准的首部

持久连接

Web 客户端经常会连接到同一个站点。比如一个页面中有很多张图片，而这些图片的一般会连接到同一个站点。因此，在第一次 HTTP 请求后，通常会保留连接。这种在事务结束后仍然保持打开状态的 TCP 连接为持久连接。

Keep-Alive

keep-alive 已经不再使用了，在最新 HTTP/1.1中也没有对它说明，但是浏览器和服务器对 keep-alive 握手的使用依旧非常广泛。 keep-alive 在早些的 RFC2068 中依旧有解释。

实现 Keep-Alive

实现 keep-alive 很简单，只要在首部包含 connection：keep-alive 就行了
如果服务器允许建立持久连接，那么只要在响应中包含 connection：keep-alive 就行了

Keep-Alive 选项

用 keep-alive 选项调节持久连接行为：
a. timeout 是在响应首部发送的，估计了服务器希望保持活跃的状态的时间（单位秒），不是一个承诺值。
b. max 是在响应首部发送的，估计了服务器希望保持活跃的最大事务数，也不是一个承诺值。

以上表示服务器最多为另外5个事务保持连接的打开状态，或者保持2分钟。
注意，Connection:Keep-Alive 首部必须随着所有希望保持连接的报文一起发送，如果没有发送，服务器将直接关闭。

剩余的每一行代表着一个首部。图中有：

  1. Host，表示服务器地址为191.101.11.227
  2. Cache-Control，缓存设置
  3. Connection,表示使用持久连接
  4. Accept,表示本机能处理 html，xhtml，xml，*(所有类型)，优先级从左到右，q 是权重系数，默认是1
  5. User-Agent,浏览器信息
  6. Accept-Language，处理的语言，为中文
  7. Accept-Encoding，表示本机能用 gzip 和 defalte进行编码。
  8. DNT,是禁止追踪，1代表用户不想被第三方网站追踪，0代表接受追踪，null代表用户不置可否

从上图可以看到

对应的响应，第一行有了200状态码，表示处理完成，一切 OK。
后面首部分别为：

  1. Date,时间
  2. Server,服务器信息，是 Apache，建立在 Ubuntu 上
  3. Last-Modified，这个实体最后修改的时间
  4. ETag，实体标记，为 gzip
  5. Accept-Ranges，服务器可接受的范围类型，为字节
  6. Vary，表示服务器会依据 Accept-Encoding 中的内容挑选出合适的类型发送
  7. Content-Encoding，内容为 gzip 编码
  8. Content-Length，内容长度
  9. Keep-Alive,持久连接设置
  10. Connection，设置连接位长连接
  11. Content-type，内容为 html 格式
